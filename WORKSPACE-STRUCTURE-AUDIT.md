# WORKSPACE-STRUCTURE-AUDIT.md

## 结论

当前工作区的主要混乱不在 Git 主线，而在以下三类边界：

1. 正式技能区 vs 候选技能区
2. 写作技能的主入口 vs 路由层 vs 辅助层
3. 顶层候选目录 / 备份目录 / 写作目录的用途边界

优先建议先清理技能入口，再处理候选区说明，最后整理写作目录边界。

---

## 发现的问题

### P1：`candidate-skills/` 与 `skills/` 存在重复并存

当前在 `candidate-skills/` 与 `skills/` 中同时存在以下同名技能：

- `novel-writer-structure`
- `novel-writing`
- `open-novel-writing`
- `writers-room-story-engine`

抽查结果显示，这些候选版与正式版前部内容基本一致，当前更像重复存放，而不是明显不同版本。

**影响：**
- 不清楚哪个目录才是正式生效版本
- 后续维护时容易改错位置
- 增加认知噪音和目录复杂度

**建议：**
- 保留 `skills/` 作为正式区
- 将 `candidate-skills/` 仅保留真正未定稿或未启用的内容
- 对重复项做清理或迁移说明

---

### P1：写作技能入口边界仍不够清晰

涉及的主要技能：

- `writers-room-story-engine`
- `open-novel-writing`
- `novel-writing`
- `novel-writer-structure`

当前问题不只是功能重叠，而是“谁是主入口、谁是路由器、谁是深加工器”还没有被固定下来。

**影响：**
- 从零开始写作时，入口不够明确
- 一个任务可能同时落入多个 skill
- 文档有说明，但目录结构本身仍给人多主入口印象

**建议分工：**
- `writers-room-story-engine`：总控路由层
- `open-novel-writing`：轻量入口 / 中文创作主流程入口
- `novel-writing`：长篇工程骨架
- `novel-writer-structure`：结构化精修 / 连载防崩

---

### P2：顶层候选 / 实验目录缺少统一说明

当前顶层存在：

- `candidate-heartbeat/`
- `candidate-proactive-full/`
- `candidate-skills/`

这些目录看起来像候选方案区或实验区，但工作区顶层没有统一说明文档解释：

- 哪些目录是正式使用的
- 哪些只是实验/备选
- 哪些后续需要清理
- 哪些只供参考

**影响：**
- 初次查看工作区时不易判断哪些目录可直接使用
- 会放大整体“结构不稳”的观感

**建议：**
- 增加一个顶层说明文档，定义正式区、候选区、备份区的用途

---

### P2：`.git.broken-20260410-094339` 仍留在工作区顶层

该目录已被 `.gitignore` 忽略，不会继续进入版本控制，但仍占据工作区顶层。

**影响：**
- 顶层视觉噪音较大
- 对不了解恢复背景的人不友好
- 让工作区看起来像恢复工作未完全收尾

**建议：**
- 后续在确认不再需要后，移出工作区或归档到更明确的备份位置

---

### P3：`novel/` 与 `writing/` 的边界不清晰

当前观察到：

- `novel/` 更像单文件作品或草稿存放区
- `writing/` 更像完整写作工作台 / 项目区

但顶层没有文档解释两者的分工。

**影响：**
- 新项目落点不明确
- 后续容易形成重复存放或结构分裂

**建议：**
- 明确 `novel/` 与 `writing/` 的职责
- 最好在 README 或结构说明文档中给出一句话解释

---

## 优先级顺序

### 第一优先级（先处理）
1. `candidate-skills/` 与 `skills/` 重复项
2. 写作四大 skill 的角色边界

### 第二优先级（随后处理）
3. 顶层候选区说明文档
4. `.git.broken-20260410-094339` 的归档或迁移

### 第三优先级（最后处理）
5. `novel/` 与 `writing/` 的职责划分

---

## 最小整改路径

1. 保留 `skills/` 为正式区
2. 为 `candidate-skills/` 增加用途说明，或清理重复项
3. 固化写作四大 skill 的角色分工
4. 增加顶层结构说明文档
5. 处理遗留 `.git.broken-*` 目录
6. 明确 `novel/` 与 `writing/` 的分工
